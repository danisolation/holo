<objective>
Eliminate N+1 query patterns in rumor watchlist summary and AI context builder.

Purpose: The rumor summary endpoint currently issues 3 queries per watchlist ticker (rumor count, score averages, dominant direction). With 20 tickers, that's 60+ queries. The AI context builder issues 1-5 queries per ticker per dimension. Both must use batch queries that scale O(1) with ticker count.

Output: Refactored rumor summary using single GROUP BY aggregate, and batch context builder methods that fetch all tickers' data in one query per dimension.
</objective>

   ```python

       select(
           Rumor.ticker_id,
           func.count(Rumor.id).label("rumor_count"),
   ```

   Simpler approach for dominant direction: use a separate small query with DISTINCT ON:
   ```python
   dir_subq = (
       select(
           RumorScore.ticker_id,
           RumorScore.direction,
           func.count(RumorScore.id).label("cnt"),
       )
       .where(RumorScore.scored_date >= seven_days_ago)
       .group_by(RumorScore.ticker_id, RumorScore.direction)
       .subquery()
   )

       select(
           dir_subq.c.ticker_id,
           dir_subq.c.direction,
       )
       .distinct(dir_subq.c.ticker_id)
       .order_by(dir_subq.c.ticker_id, dir_subq.c.cnt.desc())
   )
   ```
