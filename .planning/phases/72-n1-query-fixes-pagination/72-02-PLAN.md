Output: Paginated responses on watchlist GET, rumor watchlist summary, and rumor posts endpoints with consistent sort order.
</objective>

   ```python
       total: int
       page: int
       per_page: int
   ```

**Part B — Add pagination to watchlist GET endpoint:**

1. Update `get_watchlist()` in `backend/app/api/watchlist.py`:
   - Add query params: `page: int = 1, per_page: int = 50`
   - Cap `per_page` at 100: `per_page = min(per_page, 100)`
   - Change response_model to `PaginatedWatchlistResponse`
   - Update `_get_enriched_watchlist(session, page, per_page)` to accept pagination params

2. Update `_get_enriched_watchlist()`:
   - Add a COUNT query first: `select(func.count()).select_from(UserWatchlist)` to get total
   - Add `.offset((page - 1) * per_page).limit(per_page)` to the main query
   - Ensure stable ordering: `.order_by(UserWatchlist.created_at.desc(), UserWatchlist.symbol.asc())` — add symbol as tiebreaker
   - Return `PaginatedWatchlistResponse(items=[...], total=total, page=page, per_page=per_page)`

3. Update `migrate_watchlist()` to call `_get_enriched_watchlist(session, page=1, per_page=50)` since it returns the full list after migration.

**Part C — Add pagination to rumor watchlist summary:**

1. Update `get_watchlist_rumor_summary()` in `backend/app/api/rumors.py`:
   - Add query params: `page: int = 1, per_page: int = 50`
   - Cap `per_page` at 100
   - Change response_model to `PaginatedRumorSummaryResponse`
   - Add COUNT of watchlist items for total
   - Apply `.offset((page - 1) * per_page).limit(per_page)` to the watchlist query
   - Ensure stable ordering on watchlist query: `.order_by(UserWatchlist.symbol.asc())`
   - Return paginated response

**Part D — Add pagination to rumor posts in /{symbol} endpoint:**

1. Update `get_ticker_rumors()` in `backend/app/api/rumors.py`:
   - Add query params: `page: int = 1, per_page: int = 20`
   - Cap `per_page` at 50
   - The existing `limit(20)` on posts query becomes `.limit(per_page).offset((page - 1) * per_page)`
   - Keep `.order_by(Rumor.posted_at.desc())` — already stable (timestamp + id)
   - Add total count of posts for this ticker to the response
   - Add `posts_total: int` field to `RumorScoreResponse` schema (optional field with default None for backward compat)

**IMPORTANT — Backward compatibility:** All new query params have defaults matching current behavior (page=1, per_page=50 for lists, per_page=20 for rumor posts). Existing API consumers that don't pass pagination params get the same response structure wrapped in `{items: [...], total, page, per_page}`. Frontend callers that currently expect a raw list will need to access `.items` — document this in the summary.

**Stable ordering requirements:**
- Watchlist: `ORDER BY created_at DESC, symbol ASC`
- Rumor summary: `ORDER BY symbol ASC`
- Rumor posts: `ORDER BY posted_at DESC` (already present)
