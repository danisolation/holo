**CRITICAL DESIGN NOTE:** The semaphore bounds how many HTTP requests are in-flight simultaneously. The per-request delay MUST still be honored to avoid triggering API rate limits. Pattern: each concurrent worker acquires the semaphore, makes one request, sleeps the delay, then releases.

**FireantCrawler — add concurrency=3:**

1. Add `concurrency: int = 3` to `__init__` parameters
2. Create `self.semaphore = asyncio.Semaphore(concurrency)` in `__init__`
3. Extract the per-ticker work into a helper coroutine:

```python
async def _crawl_one_ticker(
    self, client: httpx.AsyncClient, symbol: str, ticker_id: int
) -> tuple[str, int, bool]:
    """Fetch and store posts for one ticker, respecting rate limit.
    Returns (symbol, posts_stored, success)."""
    async with self.semaphore:
        try:
            posts = await self._fetch_posts(client, symbol)
            stored = await self._store_posts(ticker_id, posts)
            if posts:
                logger.debug(f"{symbol}: {len(posts)} posts fetched, {stored} new stored")
            return (symbol, stored, True)
        except Exception as e:
            logger.warning(f"Fireant crawl failed for {symbol}: {type(e).__name__}: {e}")
            return (symbol, 0, False)
        finally:
            # Rate limit delay INSIDE semaphore to prevent burst on release
            await asyncio.sleep(self.delay)
```

4. Replace the sequential for-loop in `crawl_watchlist_tickers()` with:
```python
tasks = [
    self._crawl_one_ticker(client, symbol, ticker_id)
    for symbol, ticker_id in ticker_map.items()
]
results = await asyncio.gather(*tasks)

for symbol, stored, ok in results:
    if ok:
        success += 1
        total_posts += stored
    else:
        failed += 1
        failed_symbols.append(symbol)
```

5. Remove the old sequential loop and the `if i < len(ticker_map) - 1: await asyncio.sleep(self.delay)` line

**CafeFCrawler — add concurrency=5:**

Same pattern as Fireant:

1. Add `concurrency: int = 5` to `__init__`, create `self.semaphore = asyncio.Semaphore(concurrency)`
2. Extract per-ticker work into `_crawl_one_ticker`:

```python
async def _crawl_one_ticker(
    self, client: httpx.AsyncClient, symbol: str, ticker_id: int
) -> tuple[str, int, bool]:
    """Fetch and store news for one ticker, respecting rate limit."""
    async with self.semaphore:
        try:
            articles = await self._fetch_news(client, symbol)
            stored = await self._store_articles(ticker_id, articles)
            if articles:

        except Exception as e:
            logger.warning(f"CafeF crawl failed for {symbol}: {type(e).__name__}: {e}")

```

3. Replace sequential loop in `crawl_all_tickers()` with `asyncio.gather(*tasks)` and process results

**Why concurrency=3 for Fireant / concurrency=5 for CafeF:**
- Fireant has a longer delay (1.5s) and uses auth tokens — conservative concurrency
- CafeF has shorter delay (1.0s) and is a public AJAX endpoint — slightly more aggressive
- Both still honor per-request delays, so effective throughput: Fireant ~3/(1.5s), CafeF ~5/(1.0s)

**IMPORTANT:** The delay is inside `finally` within the semaphore context to prevent all workers from releasing simultaneously and creating a burst of new requests.

**NOTE:** All 3 RSS crawlers share the same `session`. SQLAlchemy AsyncSession is NOT safe for concurrent use from multiple tasks. To fix this, each RSS crawler must get its own session:

```python
# Create separate sessions for concurrent crawlers
async with async_session() as f319_session, \
           async_session() as vnex_session, \
           async_session() as vs_session:
    f319 = F319Crawler(f319_session)
    vnexpress = VnExpressCrawler(vnex_session)
    vietstock = VietstockCrawler(vs_session)

    f319_result, vnexpress_result, vietstock_result = await asyncio.gather(
        f319.crawl_rss(ticker_map=ticker_map),
        vnexpress.crawl_rss(ticker_map=ticker_map),
        vietstock.crawl_rss(ticker_map=ticker_map),
    )
```

This is important — SQLAlchemy AsyncSession is not thread-safe or coroutine-safe for concurrent writes. Each concurrent crawler needs its own session.
