## Integration Findings

### HIGH: Frontend Signal Display Mismatch

Backend produces signal values per analysis type that the frontend doesn't color-code:

| Analysis Type | Backend Values | Frontend Maps? |
|---|---|---|
| Technical | strong_buy, buy, neutral, sell, strong_sell | ✅ Yes |
| Fundamental | strong, good, neutral, weak, critical | ❌ Gray fallback |
| Sentiment | very_positive, positive, neutral, negative, very_negative | ❌ Gray fallback |
| Combined | mua, ban, giu | ❌ Gray fallback |

**Impact:** Combined recommendation card, fundamental/sentiment cards, watchlist signal badges, and dashboard summary all render with neutral gray styling instead of meaningful green/red colors.

**Fix scope:** Add ~11 entries to `SIGNAL_CONFIG` in `analysis-card.tsx` + update `isBuy`/`isSell` checks in `watchlist-table.tsx` and `dashboard/page.tsx`.

### LOW: Indicator API Limit Cap

Backend caps indicator response at 60 rows (`min(limit, 60)`) while frontend requests 365. SMA/BB overlay lines on candlestick chart only cover ~60 trading days on 6M+ views.

**Fix:** Increase cap in `backend/app/api/analysis.py` to 500.

### INFO: Dual Watchlist Architecture

Frontend watchlist (zustand/localStorage) and Telegram watchlist (PostgreSQL `user_watchlist` table) are independent. Adding a ticker on the web doesn't add it to Telegram alerts. This is consistent with the single-user design but worth documenting.
